# Hexappic Operating Model

## Summary
Hexappic runs like a compressed three-role indie app studio.

## Roles
- **Sponsor / CEO (Zac):** sets direction and approves major bets
- **PM (Hex):** backlog, scoping, prioritization, synthesis, handoff
- **Dev:** implementation, feasibility, technical challenge
- **Research:** customer pain validation, competitor weakness, evidence quality

## Principles
- One clear owner per function
- Minimal workflow states
- Every ticket links to evidence
- Done requires acceptance evidence
- Scope is a weapon, not a compromise

## Ticket States
- Backlog
- Ready
- In Progress
- Review
- Blocked
- Done

## Required Ticket Fields
- ID
- Title
- Owner
- Status
- Source docs
- Problem
- Smallest acceptable version
- Acceptance criteria
- Evidence

## Validation Rule
A ticket cannot be considered done unless it links to acceptance evidence.
